## 二、整合冲突与决议

### 冲突 1：文档语言 → 中文

- 初版为英文，后续讨论确认团队中文沟通，AGENTS.md 作为 AI 记忆体中文更直接。
- 杜偲妍版本约定「代码注释与文档用英文」。决议：AGENTS.md 正文中文，代码注释仍英文。

**最终：AGENTS.md 全文中文，代码注释英文。**

### 冲突 2：项目定位 → 复刻优先

- 初版定位为「跨平台复刻」，但未强调行为一致性。
- 杜偲妍提出「复刻优先于重设计」原则：Mercury 原版行为是唯一事实来源，功能行为、数据语义、Agent 行为、UX 逻辑不得改变。

**最终：采纳「复刻优先于重设计」作为第 1 节核心原则。**

### 冲突 3：Reader 处理管线 → 固定流程

- 杜偲妍提出 Reader Pipeline 必须固定：Feed Entry → Raw HTML → Readability → Sanitization → Markdown → Rendered HTML → Reader View。所有步骤在 Rust Core 完成。
- 初版未明确管线顺序。

**最终：采纳固定管线，禁止 React 执行任何内容处理步骤。创建 ADR 006。**

### 冲突 4：数据库操作分层 → Repository 模式

- 杜偲妍提出严格分层：Command → Service → Repository → SQLite。禁止 Command 直接拼接 SQL。
- 初版 AI 使用指南有类似约束但未明确分层名称。

**最终：采纳 Repository 三层分层。创建 ADR 005。**

### 冲突 5：Agent 运行时 → 统一状态机

- 杜偲妍提出所有 Agent 必须实现统一状态机（Idle/Running/Succeeded/Failed/Cancelled），明确允许与禁止的状态转换。
- 初版 AI 使用指南引用了原始 Mercury 的 `AgentRunStateMachine`，但未形式化转换规则。

**最终：采纳统一状态机合约。创建 ADR 007。**

### 冲突 6：路线图 → 四阶段 vs 七周

- 初版为老师认可的四阶段递增交付模型。
- 刘欣慧版为七周线性排期。

**最终：保留四阶段递增模型（已获老师认可），七周排期作为参考不纳入。**

### 冲突 7：设计决策表达 → 含验证方式

- 刘欣慧版 8 个决策每项含「方案/理由/实现要点/验证方式」四段，比初版 ADR 表格更实操。
- 初版仅 ADR 索引表。

**最终：第 7 节用刘欣慧的详细格式展开 8 个设计决策，第 12 节保留 ADR 索引。**
